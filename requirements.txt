# Requirements Document

## Introduction

The Stock Portfolio Advisor is a system that helps users manage and evaluate their stock portfolio. It allows users to define rating preferences and custom criteria for evaluating stocks, provides live risk assessments and buy/hold/sell recommendations, and runs background agents every 30 minutes to fetch and refresh stock data. The system aims to give users a personalized, data-driven view of their portfolio health.

## Glossary

- **Advisor**: The core system responsible for evaluating stocks and generating recommendations.
- **Portfolio**: The collection of stocks a user is tracking or has invested in.
- **Preference**: A user-defined weighting or criterion used to score and rank stocks.
- **Custom_Criterion**: A user-defined rule or metric beyond the default preference set.
- **Stock_Agent**: The background process that fetches and updates stock data on a scheduled interval.
- **Risk_Score**: A computed numeric value (0–100) representing the risk level of a stock.
- **Recommendation**: An Advisor-generated signal of BUY, HOLD, or SELL for a given stock.
- **Stock_Data**: Market data for a stock including price, volume, volatility, P/E ratio, market cap, and sector.
- **User**: A person interacting with the Stock Portfolio Advisor.
- **LLM_Agent**: The LangChain ReAct-style agent that reasons over enriched data and produces natural language recommendation rationale.
- **Rationale**: A natural language explanation generated by the LLM_Agent justifying a BUY/HOLD/SELL recommendation.
- **News_Sentiment**: Aggregated sentiment signal derived from recent news articles about a stock.
- **Earnings_Calendar**: Upcoming and recent earnings event data for a stock (dates, EPS estimates, actuals).
- **SEC_Filing**: A regulatory filing submitted to the SEC EDGAR system (e.g., 10-K, 10-Q, 8-K).
- **Refresh_Interval**: The configurable time period (in minutes) between background agent refresh cycles.
- **BaseAdapter**: The abstract interface that all data source adapters must implement to integrate with the tool registry.
- **ToolRegistry**: The component responsible for loading, instantiating, and exposing active LangChain tools from configuration.
- **tools_config.yaml**: The configuration file that declares which tools are active, which adapter backs each tool, and per-tool settings.
- **Civic_Auth**: The third-party authentication provider used by this system. Civic Auth handles OAuth 2.0 / PKCE flows and supports Google, Apple, email, and passkey sign-in. The Python SDK (`civic-auth[fastapi]`) provides FastAPI-native `get_current_user` and `require_auth` dependencies. The Civic-provided user ID (`user.get("id")`) is used as the `user_id` for all database queries.
- **ScoreBreakdown**: A structured breakdown of the normalized component scores (volatility, beta, debt-to-equity, sector), the weights actually used, and the preference and criteria adjustments that produced a Risk_Score. Returned alongside the score for frontend display.

## Requirements

### Requirement 1: User Preference Configuration

**User Story:** As a user, I want to configure how stocks are rated based on my investment preferences, so that the Advisor evaluates stocks according to my personal strategy.

#### Acceptance Criteria

1. THE Advisor SHALL provide a default set of rating preferences including risk tolerance, time horizon, sector preference, dividend preference, and growth vs. value orientation.
2. WHEN a user updates a preference, THE Advisor SHALL recalculate all stock scores and recommendations using the updated preferences within 5 seconds.
3. WHILE a user is editing preferences, THE Advisor SHALL display a real-time preview of how the changes affect the current portfolio scores.
4. IF a user submits a preference configuration with missing required fields, THEN THE Advisor SHALL return a descriptive validation error identifying each missing field.
5. THE Advisor SHALL persist user preferences across sessions.

---

### Requirement 2: Custom Criteria Management

**User Story:** As a user, I want to add custom evaluation criteria beyond the defaults, so that I can incorporate my own investment logic into stock ratings.

#### Acceptance Criteria

1. THE Advisor SHALL allow a user to define a Custom_Criterion with a name, description, numeric weight (1–10), and a comparison rule (e.g., metric greater than threshold).
2. WHEN a Custom_Criterion is saved, THE Advisor SHALL incorporate it into the scoring model for all stocks in the Portfolio.
3. WHEN a user deletes a Custom_Criterion, THE Advisor SHALL remove it from the scoring model and recalculate all affected stock scores.
4. IF a user provides an invalid Custom_Criterion (e.g., weight outside 1–10 range or missing name), THEN THE Advisor SHALL return a descriptive validation error.
5. THE Advisor SHALL support a minimum of 20 active Custom_Criteria per user.

---

### Requirement 3: Live Risk Assessment

**User Story:** As a user, I want to see live risk information for each stock in my portfolio, so that I can make informed decisions about whether to add, hold, or remove a stock.

#### Acceptance Criteria

1. THE Advisor SHALL compute a Risk_Score for each stock based on volatility, beta, debt-to-equity ratio, and sector risk weighting.
2. WHEN Stock_Data is updated, THE Advisor SHALL recompute the Risk_Score for the affected stock within 2 seconds.
3. THE Advisor SHALL display a Recommendation (BUY, HOLD, or SELL) for each stock derived from the Risk_Score and the user's active Preferences and Custom_Criteria.
4. WHEN a stock's Risk_Score crosses a user-defined threshold, THE Advisor SHALL notify the user with the stock symbol, current Risk_Score, and the triggered threshold.
5. IF Stock_Data for a stock is unavailable, THEN THE Advisor SHALL display the last known Risk_Score with a staleness indicator showing the age of the data.
6. THE Advisor SHALL display the normalized component scores (volatility, beta, debt-to-equity, sector) that contributed to the Risk_Score alongside the final score.
7. THE Advisor SHALL display the BUY/HOLD/SELL recommendation thresholds as configurable values (controlled via `SCORE_BUY_THRESHOLD` and `SCORE_SELL_THRESHOLD` environment variables).

---

### Requirement 4: Portfolio Management

**User Story:** As a user, I want to add and remove stocks from my portfolio, so that I can track the stocks relevant to my investment strategy.

#### Acceptance Criteria

1. WHEN a user adds a stock by ticker symbol, THE Advisor SHALL validate the symbol against a known securities list and add it to the Portfolio.
2. IF a user provides an unrecognized ticker symbol, THEN THE Advisor SHALL return an error message indicating the symbol was not found.
3. WHEN a stock is added to the Portfolio, THE Advisor SHALL immediately fetch Stock_Data and compute an initial Risk_Score and Recommendation.
4. WHEN a user removes a stock from the Portfolio, THE Advisor SHALL remove all associated Stock_Data, Risk_Score, and Recommendation from the active view.
5. THE Advisor SHALL support a Portfolio of up to 100 stocks per user.

---

### Requirement 5: Background Stock Data Refresh

**User Story:** As a user, I want stock data to be automatically refreshed in the background, so that my portfolio assessments stay current without manual intervention.

#### Acceptance Criteria

1. THE Stock_Agent SHALL fetch updated Stock_Data for all stocks in all active Portfolios at the configured Refresh_Interval (default 30 minutes), running the data pipeline and LLM_Agent cycle sequentially within the same refresh job.
2. WHEN the Stock_Agent completes a refresh cycle, THE Advisor SHALL update all Risk_Scores and Recommendations based on the new Stock_Data.
3. IF the Stock_Agent fails to fetch Stock_Data for a stock during a refresh cycle, THEN THE Stock_Agent SHALL retry the fetch up to 3 times before marking the stock data as stale.
4. THE Stock_Agent SHALL log the start time, end time, number of stocks updated, and any errors for each refresh cycle.
5. WHILE a refresh cycle is in progress, THE Advisor SHALL continue serving the most recently available Stock_Data to users without interruption.
6. THE Stock_Agent SHALL support concurrent fetching of Stock_Data for multiple stocks to complete a full refresh cycle within 5 minutes for a Portfolio of 100 stocks.
7. THE Refresh_Interval SHALL be configurable via the `REFRESH_INTERVAL_MINUTES` environment variable, and changes SHALL take effect on the next scheduler restart without requiring a code change.
8. THE LLM_Agent cycle SHALL only begin after the data pipeline cycle has completed for the same refresh interval, ensuring the LLM_Agent always reasons over the most recently fetched Stock_Data.

---

### Requirement 6: Stock Data Parsing and Serialization

**User Story:** As a developer, I want stock data to be reliably parsed and serialized, so that data integrity is maintained across fetches, storage, and display.

#### Acceptance Criteria

1. WHEN the Stock_Agent receives a Stock_Data payload from an external source, THE Stock_Agent SHALL parse it into a structured Stock_Data object.
2. IF a Stock_Data payload is malformed or missing required fields, THEN THE Stock_Agent SHALL reject the payload and log a descriptive parse error.
3. THE Advisor SHALL serialize Stock_Data objects into a storage format for persistence.
4. FOR ALL valid Stock_Data objects, parsing then serializing then parsing SHALL produce an equivalent Stock_Data object (round-trip property).

---

### Requirement 7: LLM-Generated Recommendation Rationale

**User Story:** As a user, I want to see a natural language explanation for each BUY/HOLD/SELL recommendation, so that I can understand the reasoning behind the Advisor's signals.

#### Acceptance Criteria

1. WHEN the LLM_Agent completes its reasoning loop for a stock, THE Advisor SHALL store a Rationale alongside the numeric Risk_Score and Recommendation.
2. THE Rationale SHALL be a non-empty natural language string that references the specific data signals (news sentiment, earnings, SEC filings, risk score) that influenced the recommendation.
3. WHEN a user requests the score for a stock, THE Advisor SHALL return the Rationale together with the Risk_Score and Recommendation in the same response.
4. IF the LLM_Agent fails to produce a Rationale (e.g., API timeout or model error), THEN THE Advisor SHALL store a fallback message indicating that the rationale is temporarily unavailable, and the numeric Risk_Score and Recommendation SHALL still be served.
5. WHEN a new Rationale is generated for a stock, THE Advisor SHALL push a `rationale_update` WebSocket event to all connected clients tracking that stock.

---

### Requirement 8: News Sentiment Integration

**User Story:** As a user, I want the Advisor to incorporate recent news sentiment into its analysis, so that recommendations reflect current market narratives.

#### Acceptance Criteria

1. THE LLM_Agent SHALL fetch recent news articles or sentiment scores for each stock from NewsAPI or Finnhub (free tier) during each refresh cycle.
2. WHEN news data is fetched, THE LLM_Agent SHALL extract a sentiment signal (positive, neutral, or negative) and a summary of the top headlines for each stock.
3. IF news data is unavailable for a stock (e.g., API limit reached or no articles found), THEN THE LLM_Agent SHALL proceed with the remaining data sources and note the absence of news data in the Rationale.
4. THE news sentiment signal SHALL be passed as a tool input to the LangChain ReAct agent for inclusion in its reasoning.

---

### Requirement 9: Earnings Calendar Integration

**User Story:** As a user, I want the Advisor to factor in upcoming and recent earnings events, so that recommendations account for earnings-driven volatility.

#### Acceptance Criteria

1. THE LLM_Agent SHALL fetch earnings calendar data for each stock via yfinance during each refresh cycle.
2. WHEN earnings data is available, THE LLM_Agent SHALL extract the next earnings date, the most recent EPS estimate, and the most recent EPS actual (if reported).
3. THE earnings data SHALL be passed as a tool input to the LangChain ReAct agent for inclusion in its reasoning.
4. IF earnings data is unavailable for a stock, THEN THE LLM_Agent SHALL proceed without it and note the absence in the Rationale.

---

### Requirement 10: SEC Filings Integration

**User Story:** As a user, I want the Advisor to consider recent SEC filings, so that recommendations reflect material regulatory disclosures.

#### Acceptance Criteria

1. THE LLM_Agent SHALL query the SEC EDGAR API for the most recent filings (10-K, 10-Q, 8-K) for each stock during each refresh cycle.
2. WHEN SEC filings are retrieved, THE LLM_Agent SHALL extract the filing type, filing date, and a brief description or summary for each recent filing.
3. THE SEC filing data SHALL be passed as a tool input to the LangChain ReAct agent for inclusion in its reasoning.
4. IF the SEC EDGAR API returns no filings or is unreachable, THEN THE LLM_Agent SHALL proceed without SEC data and note the absence in the Rationale.

---

### Requirement 11: Configurable Refresh Interval

**User Story:** As an operator, I want to configure the refresh interval via an environment variable, so that I can tune the system's data freshness without modifying code.

#### Acceptance Criteria

1. THE system SHALL read the refresh interval from the `REFRESH_INTERVAL_MINUTES` environment variable at startup.
2. IF `REFRESH_INTERVAL_MINUTES` is not set, THE system SHALL default to 30 minutes.
3. IF `REFRESH_INTERVAL_MINUTES` is set to a value that is not a positive integer, THEN THE system SHALL log a configuration error and fall back to the default of 30 minutes.
4. THE configured interval SHALL apply to both the data pipeline refresh cycle and the LLM_Agent refresh cycle so they remain synchronized.

---

### Requirement 12: Extensible Tool and Adapter Configuration

**User Story:** As a developer, I want to add or swap data source adapters without modifying core agent code, so that the system can be extended or reconfigured through configuration alone.

#### Acceptance Criteria

1. WHEN the system starts, THE ToolRegistry SHALL load the list of active tools from `tools_config.yaml` and instantiate only the adapters whose `enabled` flag is `true`.
2. WHEN a developer adds a new data source, THE system SHALL require only implementing `BaseAdapter` and adding a corresponding entry to `tools_config.yaml` — no changes to core agent code SHALL be required.
3. WHEN a tool is disabled by setting `enabled: false` in `tools_config.yaml`, THE system SHALL exclude that tool from the LangChain agent's tool list on the next startup without requiring a code change.
4. IF `tools_config.yaml` is missing, malformed, or references an unregistered adapter name, THEN THE system SHALL raise a descriptive startup error identifying the specific problem.

---

### Requirement 13: User Authentication and Authorization

**User Story:** As a user, I want to securely authenticate before accessing my portfolio data, so that my investment information is private and scoped only to me.

#### Acceptance Criteria

1. Users SHALL authenticate via Civic Auth before accessing any protected resource; unauthenticated requests to protected endpoints SHALL be rejected.
2. WHEN a user authenticates successfully, THE system SHALL create or retrieve a user record using the Civic-provided user ID (`user.get("id")`), mapping it to `user_id` in all database queries.
3. ALL portfolio, preference, score, and criteria data SHALL be scoped to the authenticated user's ID — no user SHALL be able to read or modify another user's data.
4. IF a request is made to a protected endpoint without a valid session, THEN THE system SHALL return HTTP 401 Unauthorized.
5. WebSocket connections SHALL be authenticated via a token query parameter (e.g. `ws://host/ws/{user_id}?token=...`) validated on connect, since browsers do not send cookies on WebSocket upgrades.
6. IF a WebSocket connection is attempted with an invalid or missing token, THEN THE system SHALL reject the connection with WebSocket close code 4001.
